# Кейс 3. Кредитный скоринг с Decision Tree

В кредитном скоринге важна не только точность модели, но и её прозрачность. Решение должно быть понятным: его нужно объяснить как риск-менеджеру, так и клиенту, а иногда – и регулятору. Именно поэтому в таких задачах часто используются интерпретируемые модели, такие как деревья решений.

#### Цель кейса

Построить модель, которая решает: выдавать кредит или отклонить заявку с учётом:

* поведения клиента
* финансовых показателей
* кредитной истории

и при этом даёт объяснимое решение, которое можно показать риск-менеджеру или клиенту.

#### Сценарий

У банка есть поток заявок. Для каждой заявки доступны признаки:

Финансовые:

* `monthly_income` – доход
* `loan_amount` – сумма кредита
* `debt_to_income` – отношение долга к доходу

Поведенческие:

* `credit_history_length` – длина кредитной истории (лет)
* `late_payments` – количество просрочек
* `credit_cards` – количество кредитных карт

Анкетные:

* `age` – возраст
* `employment_years` – годы занятости
* `owns_home` (0/1) – владеет ли домом

Задача:

→ классифицировать: `"approve"` или `"reject"`

#### Подготовка данных

```php
use Rubix\ML\Datasets\Labeled;

$samples = [
    // income, loan, dti, history, late, cards, age, job, home
    [5000, 10000, 0.2, 5, 0, 2, 35, 5, 1],
    [3000, 15000, 0.6, 2, 3, 4, 28, 2, 0],
    [8000, 20000, 0.25, 8, 0, 3, 45, 10, 1],
    [2500, 5000, 0.5, 1, 2, 1, 23, 1, 0],
    [6000, 12000, 0.3, 6, 1, 2, 38, 7, 1],
    [4000, 18000, 0.7, 3, 4, 5, 30, 3, 0],
];

$labels = [
    'approve',
    'reject',
    'approve',
    'reject',
    'approve',
    'reject'
];

$dataset = new Labeled($samples, $labels);
```

#### Обучение модели

Здесь уже имеет смысл задать больше параметров, приближенных к реальности.

```php
use Rubix\ML\Classifiers\ClassificationTree;

$tree = new ClassificationTree(
    maxHeight: 10,
    maxLeafSize: 2
);

$tree->train($dataset);
```

**Что означают параметры**

* `maxHeight` – количество уровней (разветвлений), которые может иметь дерево
* `maxLeafSize` – минимальное количество объектов в листе (если в узле остаётся меньше, разбиение прекращается)

Это уже реальные рычаги управления: переобучением, стабильностью модели и интерпретируемостью.

#### Предсказание

```php
$applicant = [[
    4500,   // income
    12000,  // loan
    0.35,   // dti
    4,      // history
    1,      // late payments
    2,      // cards
    32,     // age
    4,      // job years
    1       // owns home
]];

$dataset = new Unlabeled([$applicant]);

$prediction = $tree->predict($dataset);
```

#### Добавим объяснимость (ключевой момент)

В реальном продукте важно не просто решение, а почему оно принято.

Простейший вариант (ручные правила, не связанные напрямую с деревом):

```php
function explainDecision(array $input): string {
    [
        $income,
        $loan,
        $dti,
        $history,
        $late,
        $cards,
        $age,
        $job,
        $home
    ] = $input;

    $reasons = [];

    if ($dti > 0.5) {
        $reasons[] = 'высокая долговая нагрузка';
    }

    if ($late > 2) {
        $reasons[] = 'частые просрочки';
    }

    if ($history < 2) {
        $reasons[] = 'короткая кредитная история';
    }

    if (empty($reasons)) {
        return 'стабильный профиль заемщика';
    }

    return implode(', ', $reasons);
}

$result = [
    'decision' => $prediction[0],
    'explanation' => explainDecision($applicant[0]),
];

echo "Решение: \n";
echo print_r($result, 1);
```

**Результат**

```
Решение: 
Array (
   [decision] => approve
   [explanation] => стабильный профиль заемщика
)
```

**Объяснение**

Результат состоит из двух частей: предсказание дерева (approve/reject) и текстовое объяснение. В реальном продукте часто важно показывать не только итоговое решение модели, но и понятную человеку причину, почему заявка выглядит рискованной.

Да, это упрощение и не отражает реальную логику модели. Но оно показывает ключевую идею: дерево решений легко "перевести" в язык бизнес-правил.

#### Как это выглядит в реальной системе

В продакшене добавляется:

* сохранение модели (serialize / storage)
* пайплайн подготовки данных
* логирование решений
* аудит (почему отказали)
* периодическое переобучение

И главное:

> возможность показать решение человеку

#### Почему здесь выигрывает дерево

В кредитном скоринге на первый план выходит не только точность модели, но и её объяснимость. Решения должны быть прозрачными – их проверяют риск-менеджеры, аудиторы и регуляторы, поэтому важно понимать, по каким правилам модель пришла к выводу. Кроме того, бизнесу нужен контроль: возможность ограничивать поведение модели и задавать понятные рамки.

Именно по этим причинам деревья решений так широко используются на практике – как самостоятельно, так и в составе более сложных моделей. Даже если более сложные модели показывают более высокую точность, они часто проигрывают в удобстве объяснения и внедрения в реальные процессы.

#### Сравнение с логистической регрессией

Ту же задачу можно решить с помощью логистической регрессии (что мы и делали в кейсе "[Одобрение кредита](../../../chast-iii.-klassifikaciya-i-veroyatnosti/logisticheskaya-regressiya/prakticheskie-keisy/keis-5.-odobrenie-kredita.md)" из главы о логистической регрессии):

```php
use Rubix\ML\Classifiers\LogisticRegression;

$model = new LogisticRegression();
$model->train($dataset);

$prediction = $model->predict($applicant);
```

При этом логистическая регрессия тоже считается интерпретируемой моделью – её можно объяснить через веса признаков, но такие объяснения менее интуитивны для бизнеса.

Логистическая регрессия считает взвешенную сумму признаков и даёт "сглаженное" решение – все факторы влияют одновременно. Даже высокий риск по одному параметру может быть компенсирован другими.

Дерево решений работает иначе: оно задаёт последовательные вопросы и использует жёсткие пороги.

Пример:

* дерево: "если `debt_to_income` > 0.5 → отказ"
* логистическая регрессия отвечает: "насколько рискован клиент" (через комбинацию факторов и их весов)
* дерево решений отвечает: "по какому правилу принято решение" (через последовательность условий)

В задачах вроде кредитного скоринга это критично:

* логистическая регрессия отвечает: "насколько рискован клиент"
* дерево решений отвечает: "по какому правилу принято решение"

Именно поэтому дерево часто выбирают там, где важна объяснимость, даже если другая модель даёт немного лучшую метрику.

### Выводы

Этот кейс хорошо иллюстрирует реальную ценность decision tree. Это уже не просто учебный алгоритм, а полноценный инструмент принятия решений, который можно объяснить, контролировать и без лишних сложностей внедрить в бизнес-процессы. Именно поэтому деревья решений остаются одним из самых практичных подходов в индустрии – особенно в задачах, где важно понимать не только результат, но и причины, которые к нему привели.
