# Кейс 1. Внутренний корпоративный ассистент по регламентам и политикам

#### Цель кейса

В этом кейсе мы построим простую RAG-систему на чистом PHP, которая позволяет сотрудникам задавать вопросы по внутренней документации компании и получать ответы на основе реальных документов.

Главная цель – показать, как RAG превращает обычное хранилище файлов и инструкций в интеллектуальный интерфейс знаний.

При этом важно не просто получить "похожий ответ" от LLM. Система должна:

* использовать только утверждённые документы компании
* не додумывать отсутствующую информацию
* сохранять связь ответа с исходными материалами
* позволять обновлять знания без переобучения модели

Этот сценарий показывает один из самых распространённых вариантов применения RAG: корпоративный ассистент знаний.

#### Сценарий

Представим компанию, где накопилось большое количество внутренних документов:

* HR-политики
* регламенты безопасности
* инструкции для сотрудников
* правила работы с корпоративными сервисами

Со временем таких документов становится слишком много. Сотрудники уже не знают:

* где искать нужную информацию
* какая версия документа актуальна
* какой именно раздел отвечает на вопрос

Вместо ручного поиска сотрудник обращается к ассистенту:

> "Можно ли работать удалённо из другой страны?"

или:

> "Через сколько дней нужно сообщить об утечке данных?"

Система должна найти соответствующие разделы документов и сформировать понятный ответ.

#### Почему здесь нужен RAG

На первый взгляд кажется, что можно просто отправить документы в LLM. Но такой подход плохо подходит для корпоративных данных.

**Документы часто обновляются**

Политики компании меняются:

* появляются новые правила
* меняются требования безопасности
* обновляются процессы

Переобучать модель после каждого изменения невозможно.

RAG решает эту проблему иначе:

```
новый документ → новый embedding → обновлённый индекс
```

Модель остаётся той же.

**Данные чувствительные**

Корпоративные документы часто содержат внутреннюю информацию:

* процессы компании
* правила безопасности
* технические детали

Передавать такие данные для обучения модели нельзя и не нужно.

RAG работает по другому принципу: документы остаются внешним источником знаний, а LLM получает только необходимый контекст во время запроса.

#### Ответ должен быть основан на конкретном тексте

Для корпоративного ассистента недостаточно "правдоподобного" ответа.

Например:

Вопрос:

> "Можно ли работать из другой страны?"

Нельзя получить ответ:

> "Обычно компании разрешают удалённую работу…"

Такой ответ может быть логичным, но бесполезным.

Нужно:

> "Согласно HR Policy v3, международная удалённая работа требует предварительного согласования."

То есть ответ должен идти из документа.

#### Архитектура решения

В этом кейсе мы реализуем RAG вручную на чистом PHP.

Общий пайплайн:

```
Документы
    ↓
Разбиение на чанки
    ↓
Создание embeddings
    ↓
Хранение векторов
    ↓
Поиск похожих фрагментов
    ↓
Формирование контекста
    ↓
LLM
    ↓
Ответ сотруднику
```

#### Этап 1. Подготовка документов

Допустим, у нас есть два документа:

HR Policy v3:

```
Удалённая работа из другой страны возможна только после согласования с HR и руководителем.
```

Security Policy v2:

```
Об инцидентах безопасности необходимо сообщить в течение 24 часов после обнаружения.
```

Но в реальной системе документы будут намного больше.

Поэтому первым шагом является разбиение текста на чанки.

**Chunking**

Большие документы нельзя передавать модели целиком.

Например:

100 страниц политики безопасности

↓

разбиваются на небольшие фрагменты:

```
Chunk 1:
Правила удалённой работы...

Chunk 2:
Процедура обработки инцидентов...

Chunk 3:
Требования к доступу...
```

Типичная стратегия:

* размер чанка: 300–400 токенов
* небольшой overlap между фрагментами
* сохранение метаданных документа

Например:

```php
[
    'document' => 'HR Policy v3',
    'content' => 'Удалённая работа из другой страны возможна...',
]
```

#### Этап 2. Создание embeddings

Теперь каждый chunk превращается в числовой вектор.

Например:

```
"Удалённая работа из другой страны..."
↓
[0.124, 0.532, 0.871 ...]
```

Этот вектор представляет смысл текста.

В итоге мы получаем:

```
HR Policy v3
    ↓
embedding

Security Policy v2
    ↓
embedding
```

#### Этап 3. Retrieval

Пользователь задаёт вопрос:

```
Можно ли работать удалённо из другой страны?
```

Система создаёт embedding запроса:

```
Question
    ↓
vector
```

Затем сравнивает его с эмбеддингами документов.

Используется косинусное сходство:

$$
similarity(A,B)= \frac{A \cdot B} {|A||B|}
$$

Чем ближе значение к 1, тем похожее содержание.

После поиска берём Top-K результатов.

Например:

```
Top-K = 3
```

Получаем:

```
[1] HR Policy v3
Удалённая работа из другой страны...

[2] Remote Work Guide
Процесс согласования...

[3] Security Policy
Требования доступа...
```

#### Этап 4. Context building

Это самый важный этап.

Мы не отправляем найденные куски напрямую в модель.

Сначала строим контролируемый контекст.

Например:

```
Ты внутренний корпоративный ассистент.
Отвечай строго на основе контекста ниже.
Если ответа нет – скажи, что информации недостаточно.

Контекст:
[HR Policy v3]
Удалённая работа из другой страны возможна только после согласования...
[Security Policy v2]
Инциденты безопасности необходимо сообщать в течение 24 часов...

Вопрос:
Можно ли работать удалённо из другой страны?
```

Почему это важно:

LLM получает не просто текст, а инструкцию:

* какие данные использовать
* чего нельзя делать
* когда нужно отказаться от ответа

Именно здесь RAG становится контролируемой системой.

#### Этап 5. Генерация ответа

Теперь LLM получает:

* инструкцию
* найденные документы
* вопрос пользователя

Она не ищет информацию сама.

Она выполняет роль преобразователя:

```
Контекст
    ↓
Понятный ответ
```

Например:

Ответ:

> Согласно HR Policy v3, работа из другой страны возможна только после предварительного согласования с HR и руководителем

#### Упрощённая реализация на PHP

Пример логики извлечения (retrieval):

```php
$queryEmbedding = createEmbedding(
    "Можно ли работать удалённо из другой страны?"
);

$scores = [];

foreach ($documents as $document) {
    $score = cosineSimilarity($queryEmbedding, $document['embedding'])
    $scores[] = ['document' => $document, 'score' => $score];
}

usort($scores, function ($a, $b) {
    return $b['score'] <=> $a['score'];
});

$topChunks = array_slice($scores, 0, 3);
```

После этого строится контекст:

```php
$context = "
Ты внутренний корпоративный ассистент.
Отвечай только на основе контекста.

Контекст:
";

foreach ($topChunks as $chunk) {
    $context .= "\n";
    $context .= "[" . $chunk['document']['name'] . "]";
    $context .= "\n";
    $context .= $chunk['document']['content'];
}
```

#### Что показывает этот кейс

Этот пример демонстрирует фундаментальную идею RAG: LLM не заменяет базу знаний.

Она становится удобным интерфейсом доступа к ней.

Вместо:

> "найди документ → прочитай → пойми → сформулируй"

сотрудник получает:

> "задай вопрос → получи ответ на основе документов"

#### Практические выводы

В этом кейсе мы реализовали RAG без фреймворков и увидели все основные этапы:

* документы превращаются в чанки
* чанки превращаются в эмбеддинги
* retrieval выбирает релевантные знания
* context building ограничивает поведение модели
* LLM формирует итоговый ответ

**Главный вывод:**

Качество корпоративного ассистента определяется не только моделью, а тем, насколько хорошо организован слой поиска и подготовки контекста.

RAG здесь выступает не как "чат с документами", а как новый интерфейс к корпоративной базе знаний.
