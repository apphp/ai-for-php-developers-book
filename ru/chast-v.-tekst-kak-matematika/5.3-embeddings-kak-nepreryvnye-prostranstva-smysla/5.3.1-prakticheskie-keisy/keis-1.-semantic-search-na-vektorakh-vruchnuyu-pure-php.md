# Кейс 1. Semantic search на векторах вручную (pure PHP)

#### Цель кейса

Показать, что [semantic search](../../../vvedenie/glossarii.md#semantic-search) – это не "магия нейросетей" и не сложная AI-инфраструктура, а обычная геометрия и несколько математических операций над векторами.

После этого кейса вы должны понять три ключевые идеи:

* embedding – это просто вектор чисел
* семантический поиск (semantic search) – это поиск ближайших [embedding-векторов](../../../vvedenie/glossarii.md#embedding-vektor) по метрике сходства
* большая часть AI-систем строится вокруг достаточно простой математики

Это один из ключевых моментов для понимания embeddings: они перестают выглядеть как "магия AI" и начинают восприниматься как прикладная математика.

#### Сценарий

Представим, что мы разрабатываем небольшую AI-функцию для внутренней базы знаний support-команды.

У нас есть набор документов:

* статьи по email-рассылкам
* материалы по Laravel
* статьи по DevOps
* документация по очередям
* несколько нерелевантных документов

Пользователь вводит запрос:&#x20;

> "ускорить отправку email"

Вместо поиск по ключевым словам мы попробуем выполнить семантический поиск, то есть поиск по смысловой близости.

Для простоты embeddings будут заданы вручную. Это нормально для учебного примера: сейчас нас интересует не качество модели, а сама механика semantic search.

**Шаг 2. Реализуем cosine similarity**

Для измерения смысловой близости используем cosine similarity.

```php
function cosineSimilarity(array $a, array $b): float {
    $dotProduct = 0.0;
    $normA = 0.0;
    $normB = 0.0;

    foreach ($a as $i => $value) {
        $dotProduct += $value * $b[$i];

        $normA += $value ** 2;
        $normB += $b[$i] ** 2;
    }

    return $dotProduct / (sqrt($normA) * sqrt($normB));
}
```

Эта функция вычисляет cosine similarity – косинус угла между векторами. Cosine similarity особенно популярна для embeddings, потому что сравнивает направление векторов, а не их абсолютную длину.

В embedding-пространстве близкие по смыслу объекты обычно имеют близкие направления векторов – этим мы и воспользуемся.

**Шаг 3. Считаем схожесть для всех документов**

```php
foreach ($documents as &$document) {
    $document['score'] = cosineSimilarity($query['embedding'], $document['embedding']);
}
```

После этого у каждого документа появляется score – оценка смысловой близости к запросу.

**Шаг 4. Сортируем результаты**

```php
usort($documents, function ($a, $b) {
    return $b['score'] <=> $a['score'];
});
```

**Шаг 5. Выводим top-N результатов**

```php
$topN = 5;

foreach (array_slice($documents, 0, $topN) as $document) {
    echo $document['title'] . ' => ' . round($document['score'], 3) . PHP_EOL;
}
```

#### Что здесь произошло на самом деле

Это самый важный момент кейса.

Мы:

* не сравнивали слова
* не использовали регулярные выражения
* не искали keyword match
* не писали правила вида `if text contains "email"`

Сам алгоритм поиска не знает, что такое "email", "Redis" или "RabbitMQ". Он работает только с embedding-векторами и их взаимным расположением.

Семантический смысл уже закодирован в embeddings на этапе их генерации моделью. Сам semantic search использует только геометрию этих векторов. Каждый embedding можно рассматривать как точку в многомерном пространстве, а semantic search сводится к поиску ближайших соседей.

На уровне базового алгоритма вся логика поиска выглядит так:

```
цикл → cosine similarity → сортировка
```

Именно в этот момент embeddings обычно перестают выглядеть как "магия AI" и начинают восприниматься как прикладная математика и геометрия многомерных пространств.

#### Почему это уже похоже на production

Хотя пример учебный, архитектурно он очень близок к реальным AI-системам.

В production изменятся:

* способ генерации embeddings
* размерность векторов
* количество документов
* инфраструктура хранения
* скорость поиска

Но базовая идея pipeline останется той же:

```
текст → embedding → similarity → nearest neighbors
```

В production вместо полного перебора обычно используются ANN-индексы (HNSW, IVF, PQ и др.), но сама идея поиска ближайших соседей остаётся неизменной.

#### Где такой подход используется

Точно такой же механизм лежит в основе:&#x20;

* семантического поиска
* RAG-систем
* AI-ассистентов
* поиска по support-тикетам
* поиска по документации
* recommendation systems
* поиска похожих событий и логов
* vector databases
* ANN-индексов и nearest neighbor search

Разница только в масштабе и инфраструктуре.

#### Ограничения учебного примера

В этом кейсе embeddings заданы вручную. В реальных системах их генерируют embedding-модели:

* OpenAI Embeddings
* Sentence Transformers
* BGE
* E5
* Instructor
* TransformersPHP
* и другие модели

Кроме того, настоящие embeddings обычно содержат сотни или тысячи измерений. Но с точки зрения алгоритма ничего принципиально не меняется.

#### Выводы

В этом кейсе мы реализовали семантический поиск практически "с нуля", используя только PHP и базовую математику.

Главная идея embeddings заключается не в сложных нейросетях, а в представлении объектов как точек непрерывного пространства.

После этого поиск превращается в геометрическую задачу: найти ближайшие точки по выбранной метрике сходства. Именно поэтому embeddings стали фундаментом современных AI-систем, семантического поиска и retrieval-архитектур.
