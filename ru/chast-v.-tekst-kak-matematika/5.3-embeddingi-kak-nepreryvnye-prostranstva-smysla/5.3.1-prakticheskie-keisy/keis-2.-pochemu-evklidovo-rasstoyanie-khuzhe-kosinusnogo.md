# Кейс 2. Почему евклидово расстояние хуже косинусного

#### Цель кейса

Показать, почему в задачах семантического поиска косинусное сходство часто оказывается предпочтительнее евклидова расстояния для ненормализованных эмбеддингов.

Это один из самых недооцененных моментов во всей теме семантического поиска. Многие разработчики механически используют косинусное сходство, не понимая, какую проблему она решает.

После этого кейса читатель должен понять:

* эмбеддинги сравнивают не только по расстоянию, но и по направлению
* длина вектора может искажать результаты поиска
* косинусное сходство лучше отражает смысловую близость
* нормализация эмбеддингов – это не случайная оптимизация, а важная часть геометрии пространства

Этот кейс особенно ценен тем, что учит не повторять готовые рецепты, а понимать их математическую причину.

#### Сценарий

Возьмем почти те же эмбеддинги, что и в предыдущем кейсе.

Представим, что пользователь снова ищет:&#x20;

> "ускорить отправку email"

Но теперь мы хотим понять:

Какую метрику расстояния лучше использовать?

Сравним два подхода:

* евклидово расстояние (euclidean distance)
* косинусное сходство (cosine similarity)

#### Почему возникает проблема

На первый взгляд кажется логичным: если векторы близко расположены – значит тексты похожи.

Но эмбеддинги работают немного тоньше.

Для них часто важнее:

* не абсолютное положение
* а направление вектора

Именно здесь евклидова метрика начинает давать странные результаты.

### Подготавливаем эмбеддинги

Добавим специальный документ с большим масштабом вектора.

```php
$documents = [
    [
        'title' => 'Оптимизация email-рассылки',
        'embedding' => [0.10, 0.84, 0.29, 0.58, 0.69],
    ],
    [
        'title' => 'Scaling PHP workers',
        'embedding' => [0.11, 0.81, 0.36, 0.63, 0.72],
    ],
    [
        'title' => 'Мониторинг email delivery',
        'embedding' => [0.09, 0.91, 0.28, 0.57, 0.74],
    ],
    
    // Тот же смысл, но другой масштаб
    [
        'title' => 'Масштабированный email embedding',
        'embedding' => [1.0, 8.4, 2.9, 5.8, 6.9],
    ],

    [
        'title' => 'Как приготовить кофе',
        'embedding' => [0.91, 0.04, 0.15, 0.08, 0.02],
    ],
];

$query = [
    'text' => 'ускорить отправку email',
    'embedding' => [0.10, 0.82, 0.31, 0.60, 0.70],
];
```

Обратите внимание:

```
Масштабированный email embedding
```

имеет почти то же направление, что и обычный email-документ, но длина вектора в 10 раз больше.

С точки зрения смысла документы почти идентичны.

Но как это воспримут разные метрики?

#### Часть 1. Евклидово расстояние

**Реализуем euclidean distance**

```php
function euclideanDistance(array $a, array $b): float {
    $sum = 0.0;

    foreach ($a as $i => $value) {
        $sum += ($value - $b[$i]) ** 2;
    }

    return sqrt($sum);
}
```

**Считаем расстояния**

```php
foreach ($documents as &$document) {
    $document['euclidean'] = euclideanDistance(
        $query['embedding'],
        $document['embedding']
    );
}
```

**Сортируем по расстоянию**

Чем меньше расстояние – тем лучше.

```php
usort($documents, function ($a, $b) {
    return $a['euclidean'] <=> $b['euclidean'];
});
```

**Результат**

Примерный вывод:

```
Оптимизация email-рассылки => 0.03
Scaling PHP workers => 0.08
Мониторинг email delivery => 0.10
Как приготовить кофе => 1.40
Масштабированный email embedding => 11.9
```

#### Что пошло не так

Это очень важный момент.

Документ:

```
Масштабированный email embedding
```

по смыслу почти идентичен запросу.

Но евклидово расстояние считает его очень далеким объектом только потому, что длина вектора намного больше.

Метрика начинает путать:

* смысл
* и масштаб

Это фундаментальная проблема евклидова расстояния для ненормализованных эмбеддингов.

#### Часть 2. Косинусное сходство

**Реализуем cosine similarity**

Теперь сравним тот же набор данных через cosine similarity.

```php
function cosineSimilarity(array $a, array $b): float {
    $dotProduct = 0.0;
    $normA = 0.0;
    $normB = 0.0;

    foreach ($a as $i => $value) {
        $dotProduct += $value * $b[$i];

        $normA += $value ** 2;
        $normB += $b[$i] ** 2;
    }

    return $dotProduct / (
        sqrt($normA) * sqrt($normB)
    );
}
```

**Считаем similarity**

```php
foreach ($documents as &$document) {
    $document['cosine'] = cosineSimilarity(
        $query['embedding'],
        $document['embedding']
    );
}
```

**Сортируем результаты**

Теперь наоборот: чем сходство больше – тем лучше.

```php
usort($documents, function ($a, $b) {
    return $b['cosine'] <=> $a['cosine'];
});
```

**Результат**

После расчета получим примерно такую картину:

```
Масштабированный email embedding => 0.999
Оптимизация email-рассылки => 0.999
Мониторинг email delivery => 0.998
Scaling PHP workers => 0.996
Как приготовить кофе => 0.218
```

#### Почему косинусное сходство работает лучше

Косинусное сходство измеряет не абсолютное расстояние между точками, а угол между векторами.

Это означает:

* длина вектора не влияет на результат косинусного сходства
* важным становится направление

А направление эмбеддинг-вектора как раз и несет основную семантическую информацию.

С точки зрения cosine similarity:

```
[0.1, 0.8, 0.3]
```

и

```
[1.0, 8.0, 3.0]
```

почти одинаковы, потому что направлены в одну сторону.

И это именно то поведение, которое нужно для семантического поиска.

#### Почему эмбеддинги часто нормализуют

Во многих эмбеддинг-системах векторы дополнительно нормализуют:

```
длина вектора = 1
```

После нормализации:

* все эмбеддинги лежат на поверхности единичной гиперсферы
* косинусное сходство становится еще стабильнее
* исчезает влияние масштаба
* поиск становится более устойчивым

Фактически система начинает сравнивать только направление смыслов.

> Важно понимать, что косинусное сходство уже компенсирует различия в длине векторов. Нормализация часто применяется дополнительно для ускорения поиска, упрощения вычислений и возможности использовать скалярное произведение вместо отдельного вычисления косинусного сходства.

#### Что важно понять концептуально

Этот кейс показывает очень глубокую идею.

Embeddings – это не координаты в привычном геометрическом смысле.

Нас интересует не:

```
"насколько далеко находятся точки"
```

а:

```
"смотрят ли они в одну смысловую сторону"
```

Это одна из причин, почему embedding-пространства работают настолько хорошо.

#### Где это особенно важно

Разница между косинусного сходства и евклидовым расстоянием особенно заметна для:

* семантического поиска
* векторные базы данных
* рекомендательные систем
* RAG-архитектур
* поиска похожих событий
* retrieval-систем
* AI-ассистентов

Неправильно выбранная метрика может полностью испортить качество поиска.

#### Выводы

В этом кейсе мы увидели, что евклидово расстояние на ненормализованных эмбеддингах может давать нежелательные результаты, поскольку чувствительно к длине вектора.

Косинусное сходство работает лучше, поскольку сравнивает направление векторов, а не их абсолютный масштаб.

Именно поэтому современные эмбеддинг-системы часто:

* используют косинусное сходство или эквивалентные ему метрики
* нормализуют эмбеддинги
* уделяют внимание выбору метрики в зависимости от модели и задачи
* интерпретируют смысл как направление в пространстве

Это один из тех редких случаев, где небольшая математическая деталь радикально меняет качество всей AI-системы.
