# 5.6.1 Различные способы добавить NER в PHP

Named Entity Recognition (NER) – одна из самых практичных задач NLP для бизнес-приложений. Она позволяет превратить обычный текст в структурированные данные, с которыми уже может работать бэкенд.

Для PHP-разработчика важен не только сам алгоритм NER, но и вопрос интеграции:

* где будет выполняться inference модели
* как PHP будет с ней взаимодействовать
* нужен ли отдельный ML-сервис
* можно ли запустить модель прямо внутри приложения
* когда лучше использовать классический NER, а когда LLM

В реальных системах существует несколько подходов

### NER как задача NLP

На концептуальном уровне NER выглядит достаточно просто.

Есть текст:

```
Apple signed a contract with John Smith in London.
```

Модель должна найти сущности:

```
Apple      → ORGANIZATION
John Smith → PERSON
London     → LOCATION
```

Внутри системы происходит несколько этапов:

```
Text
 ↓
Tokenizer
 ↓
Model
 ↓
Entity labels
 ↓
Structured data
```

Сначала текст разбивается на токены. Затем модель анализирует последовательность токенов и для каждого фрагмента определяет роль.

Например:

```json
{
  "entities": [
    {
      "text": "Apple",
      "type": "ORG"
    },
    {
      "text": "John Smith",
      "type": "PERSON"
    }
  ]
}
```

После этого обычное PHP-приложение уже может работать не с текстом, а с объектами:

```php
[
    [
        "type" => "PERSON",
        "value" => "John Smith"
    ],
    [
        "type" => "ORG",
        "value" => "Apple"
    ]
]
```

Например, сохранить компанию в CRM, найти пользователя, связать письмо с заказом или построить аналитику.

Главный вопрос – где будет выполняться эта модель.

### Вариант 1. spaCy через отдельный NLP-сервис

Один из самых распространенных промышленных вариантов – вынести NLP-модель в отдельный сервис.

Архитектура выглядит так:

```
PHP/Laravel
  ↓
HTTP
  ↓
Python FastAPI
  ↓
spaCy NER model
```

PHP остается основным приложением, а Python отвечает только за машинное обучение.

Например, Laravel отправляет текст:

```php
$response = Http::post(
    'http://nlp-service/entities', ['text' => $tex]
);
```

Python-сервис принимает запрос и запускает модель:

```python
doc = nlp(text)

entities = []

for entity in doc.ents:
    entities.append({
        "text": entity.text,
        "label": entity.label_
    })
```

Ответ возвращается в PHP:

```json
[
  {
    "text": "Apple",
    "label": "ORG"
  },
  {
    "text": "John Smith",
    "label": "PERSON"
  }
]
```

#### Когда выбирать spaCy

Этот подход хорошо подходит для компаний, где уже есть Python-команда и отдельная ML-инфраструктура.

Преимущества:

* зрелая экосистема
* большое количество готовых моделей
* простая интеграция с другими NLP-инструментами
* удобно масштабировать отдельно от PHP-приложения

Недостатки:

* появляется отдельный сервис
* нужно поддерживать Python-окружение
* усложняется деплой

Архитектурно это классический вариант: PHP отвечает за бизнес-логику, Python – за AI-компонент.

### Вариант 2. Transformers через Python API

Следующий уровень – использовать современные transformer-модели для NER.

Вместо классических статистических NER-моделей и моделей на основе ручных признаков используются transformer-модели:

* BERT
* RoBERTa
* DeBERTa
* специализированные NER-модели

Важно понимать, что spaCy и Transformers не являются взаимоисключающими технологиями. spaCy может использовать transformer-модели внутри своих пайплайнов через компонент spacy-transformers. Поэтому выбор часто делается не между spaCy и трансформерами, а между различными способами их использования.

Архитектура:

```
PHP
 ↓
HTTP
 ↓
Transformers pipeline
 ↓
NER model
```

Например, Python-сервис использует pipeline:

```python
pipeline("ner", model="dslim/bert-base-NER")
```

После обработки текст превращается в набор найденных сущностей:

```json
[
 {
  "word": "Apple",
  "entity": "ORG"
 }
]
```

Главное отличие от spaCy – здесь используются современные модели на основе трансформеров.

Они лучше работают там, где важен контекст.

Например:

```
Apple released a new product.
```

Здесь Apple – компания.

Но:

```
I ate an apple.
```

Здесь apple – обычный предмет.

Transformer-модель учитывает широкий контекст предложения и связи между словами, что часто позволяет точнее определять тип сущности в неоднозначных случаях.

#### Преимущества

* современные модели
* высокое качество распознавания
* огромная экосистема Hugging Fac
* можно выбирать модели под конкретную задачу.

#### Недостатки

* сложнее инфраструктура
* модели тяжелее
* требуется больше ресурсов

### Вариант 3. ONNX-модели напрямую

Для PHP-разработчика особенно интересен вариант с ONNX.

Идея простая – модель обучается в привычной ML-среде:

```
PyTorch
  ↓
export
  ↓
ONNX
  ↓
runtime
  ↓
PHP
```

После экспорта PHP больше не вызывает Python.

Модель становится обычным компонентом приложения.

Архитектура:

```
Laravel
  ↓
ONNX Runtime
  ↓
BERT NER model
```

На практике для запуска ONNX-моделей из PHP обычно используются специализированные расширения или библиотеки-обёртки над ONNX Runtime. Это позволяет выполнять inference без отдельного Python-сервиса.

Процесс обработки:

```
Text
  ↓
Tokenizer
  ↓
input_ids
attention_mask
  ↓
ONNX model
  ↓
logits
  ↓
Entity labels
```

То есть PHP сам:

1. принимает текст
2. преобразует его в токены
3. отправляет данные в ONNX Runtime
4. получает результаты модели
5. преобразует их в сущности

#### Преимущества

* меньше инфраструктуры
* часто более быстрый inference по сравнению с исходным фреймворком
* нет отдельного Python-сервиса
* модель можно встроить рядом с Laravel

#### Недостатки

* сложнее работа с tokenizer
* нужно понимать внутренний формат моделей
* больше ответственности на стороне PHP-разработчика

Этот подход особенно интересен для систем, где важны:

* низкая задержка
* автономность
* простой deployment

Например, SaaS-приложение может иметь Laravel API и локально выполнять часть AI-задач.

### Вариант 4. TransformersPHP – PHP-native подход

Если приложение полностью построено на PHP, логичный вариант – использовать PHP-библиотеки для работы с transformer-моделями.

Архитектура:

```
Laravel
 ↓
TransformersPHP
 ↓
Transformer model
 ↓
NER pipeline
```

В таком случае модель вызывается непосредственно из PHP-кода:

```php
use function Codewithkyrian\Transformers\Pipelines\pipeline;

$pipeline = pipeline(task: 'token-classification', modelName: 'Xenova/bert-base-NER');
$result = $pipeline($text);
```

Приложение не делает HTTP-запросы к внешнему сервису.

Всё происходит внутри одного runtime.&#x20;

Хотя под капотом могут использоваться ONNX Runtime, FFI или другие нативные компоненты, для разработчика модель выглядит как обычная PHP-библиотека.

#### Преимущества

* чистый PHP-стек
* проще деплой
* меньше компонентов
* удобно для PHP-команд

#### Недостатки

* экосистема меньше, чем у Python
* не все модели поддерживаются
* иногда сложнее найти готовые решения

Для классического Laravel-приложения это очень привлекательный вариант: AI-компонент становится частью обычного бэкенда.

### Вариант 5. LLM как NER

Современный вариант – использовать большие языковые модели как механизм извлечения сущностей.

Вместо отдельной NER-модели мы отправляем инструкцию:

```
Extract entities.

Text:
Apple signed a contract with John Smith.
```

LLM возвращает структурированный результат:

```json
{
  "company": "Apple",
  "person": "John Smith"
}
```

В таком подходе модель сама понимает:

* какие сущности нужны
* в каком формате их вернуть
* как интерпретировать сложный текст

Например, можно сразу попросить:

```json
{
  "customer": "",
  "company": "",
  "order_id": "",
  "priority": ""
}
```

И получить результат, готовый для бизнес-логики.

#### Преимущества

* высокая гибкость
* можно менять схему результата
* хорошо работает со сложными текстами
* во многих случаях не требуется обучать отдельную NER-модель

Для специализированных доменных сущностей (например, медицинских кодов, внутренних бизнес-терминов или отраслевых классификаторов) может потребоваться fine-tuning, few-shot prompting или использование RAG.

#### Недостатки

* стоимость запросов
* задержка
* нужен контроль качества
* возможны ошибки генерации

Поэтому LLM часто используют не вместо NER, а вместе с ним.

Например:

* быстрый ONNX NER извлекает клиентов, компании и другие именованные сущности, а даты и номера заказов дополнительно определяются правилами, регулярными выражениями или специализированными моделями
* LLM анализирует сложное описание проблемы клиента

### Какой вариант выбрать для PHP-приложения

Универсального решения нет.

Если нужен классический production NLP:

```
Laravel
 +
Python NLP service
 +
spaCy/Transformers
```

Если важна автономность:

```
Laravel
 +
ONNX Runtime
```

Если проект полностью PHP:

```
Laravel
 +
TransformersPHP
```

Если нужно быстро извлекать сложные данные:

```
Laravel
 +
LLM structured output
```

В реальных AI-системах часто используется комбинация подходов.

Например, в нашей платформе поддержки клиентов:

```
Email
 ↓
NER
 ↓
Клиент
Компания
Заказ
Дата
 ↓
CRM
 ↓
LLM response generation
```

NER отвечает за извлечение фактов, а LLM – за понимание ситуации и генерацию ответа.

Именно такое разделение позволяет строить надежные AI-приложения: каждая модель выполняет свою роль.
